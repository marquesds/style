# AI Collaboration Hygiene

Delegate acceleration, not judgment. Review AI output harder than human output.

## When Not to Delegate

Retain human authorship for:

| Category | Why |
|---|---|
| Security-sensitive code (auth, crypto, injection guards) | Hallucinated APIs create silent vulnerabilities |
| Irreversible migrations (DROP, purge, schema rename) | Confidence ≠ correctness; blast radius is total |
| Novel domain with no prior art in codebase | Model synthesizes from training data, not your domain |
| Regulatory / compliance decisions | Model cannot know current law or contractual obligations |
| Architectural choices (new service, new port, ADR-worthy) | Design tradeoffs require context the model lacks |

Delegate safely: boilerplate, test scaffolding, refactoring with a clear spec,
documentation drafts, routine CRUD.

## Reviewing AI-Generated Code Differently

AI output passes surface syntax checks and reads plausibly. The failure modes differ:

- **Hallucinated APIs**: function name exists; parameters are wrong. Verify every call
  against the installed version (skill:source-driven-development).
- **Fabricated tests**: test passes but does not pin the behavior. Introduce a deliberate
  bug in the implementation — if the test stays green, the test is worthless.
- **Hyrum's Law amplified**: model may cement an accidental behavior as a public API by
  documenting it. Be stricter about what the public surface actually is.
- **Confident wrong edges**: model handles the happy path; boundary conditions look
  plausible but have not been exercised. Run with boundary inputs.
- **Stale patterns**: training data skews toward older API versions. Check for
  deprecation warnings on every merged import.

Apply the full six-axis review (skill:code-review-and-quality). Treat AI output as
untrusted contributor code, not trusted colleague code. The same posture now
applies to **every** PR — see skill:code-review-and-quality → "PR Description Is
Untrusted" — AI-authored code is one case of a wider default.

## Architectural Decisions Stay Human

A model can propose an architecture. A human writes the ADR, reads the tradeoffs, and
owns the consequences. No ADR committed without a human signing off.

Model confidence in its own proposal is not evidence of correctness. Architectural
choices have long blast radii; a plausible-sounding wrong decision compounds with every
feature added on top.

## Citation Requirement

AI-generated code for nontrivial framework usage → require a citation to the official
docs for the installed version before merging. "The model suggested it" is not a
citation. If the citation is missing, look it up. See skill:source-driven-development.

## Prompt Injection Awareness

When the AI reads user-generated content (tickets, docs, emails) as part of its task,
that content may carry adversarial instructions. See skill:llm-prompt-injection.

## GOOD

```text
AI-generated PR checklist:
- [ ] Every framework API verified against installed version docs.
- [ ] One deliberate mutation per test — test catches it.
- [ ] Boundary inputs exercised, not just the happy path.
- [ ] ADR authored by a human if architecture changed.
- [ ] Auth / crypto code rewritten by human, not merged as-is.
```

## BAD

Merging AI-generated code because "it looks right and the tests pass." Tests generated
by the same model that wrote the implementation may not independently verify behavior.

## Red Flags

- AI-generated auth middleware merged without human rewrite.
- Test suite generated alongside the implementation with no independent verification.
- ADR with "as suggested by the model" as the decision rationale.
- Framework calls never checked against installed version docs.
- Migration script generated by AI executed on production without a DBA review.
- No `skill:requirements-crushing` brief before delegating a large feature to AI.
